# Keycloak Configuration Guide for BookStack SSO

This document describes the Keycloak configuration required to enable Single Sign-On between the OSP Search Admin platform, BookStack (Knowledge Base), and Keycloak.

For the BookStack-side deployment and configuration, see [bookstack-integration.md](./bookstack-integration.md).

---

## Resources Introduced by This Integration

| Resource type | Name | Purpose |
|---|---|---|
| Realm role | `platform-admin` | Identifies platform-level administrators; used to grant edit access in BookStack (and reusable for future cross-application roles) |
| OIDC Client | `bookstack` | The OIDC client used by BookStack to authenticate users against Keycloak |
| Protocol Mapper | `realm roles to id token` (on the `bookstack` client's dedicated scope) | Emits the user's realm roles as a top-level `roles` claim in the ID token, so BookStack can read them for role synchronisation |

---

## Prerequisites

- A running Keycloak instance (version 22 or later recommended).
- Administrator access to the realm used by OSP Search Admin.
- The BookStack application reachable at a stable hostname (for local development: `http://host.docker.internal:6875`; for production: a dedicated FQDN such as `https://kb.company.com`).

---

## 1. Create the `platform-admin` Realm Role

**Path:** Keycloak console → target realm → **Realm roles → Create role**

| Field | Value |
|---|---|
| Role name | `platform-admin` |
| Description | Platform-wide administrator — can manage Knowledge Base and other platform-level resources |

### Assign the role to platform administrators

For each user who should have platform-admin privileges:

**Users → select user → Role mapping → Assign role** → filter **Realm roles** → tick `platform-admin` → **Assign**

> `platform-admin` is a **realm role**, not a client role. This is intentional: realm roles are automatically included in the token under `realm_access.roles` without requiring a custom Protocol Mapper, and they can be reused across other applications (e.g. future Customer Portal, support tooling).

---

## 2. Create the `bookstack` OIDC Client

**Path:** Keycloak console → target realm → **Clients → Create client**

- **Client type:** OpenID Connect
- **Client ID:** `bookstack`

Proceed through the creation wizard; the detailed settings are described below.

### 2.1 Settings tab

| Field | Value |
|---|---|
| Client authentication | **ON** (confidential client) |
| Authorization | OFF |
| Standard flow | **ON** |
| Direct access grants | OFF |
| Service accounts roles | OFF |
| Root URL | `http://host.docker.internal:6875` |
| Home URL | `http://host.docker.internal:6875` |
| Valid redirect URIs | `http://host.docker.internal:6875/oidc/callback` |
| Valid post logout redirect URIs | `http://host.docker.internal:6875/*` |
| Web origins | `http://host.docker.internal:6875` |

> In production, replace every `http://host.docker.internal:6875` above with the real BookStack FQDN, e.g. `https://kb.company.com`. See the "Production Replacement Checklist" below.

### 2.2 Advanced tab — Backchannel Logout

| Field | Value |
|---|---|
| Backchannel logout URL | `http://host.docker.internal:6875/oidc/logout` |
| Backchannel logout session required | **ON** |
| Backchannel logout revoke offline sessions | ON |

This configuration ensures that when a user logs out of the OSP admin portal (which triggers a full Keycloak logout), Keycloak notifies BookStack so it can terminate the matching session.

> **⚠️ Caveat:** BookStack's OIDC implementation may not fully support backchannel logout across all versions. If you observe that BookStack sessions remain active after adminui logout, switch from **Backchannel Logout URL** to **Front-channel Logout URL** (which triggers the logout via browser iframes instead).

### 2.3 Credentials tab

| Field | Value |
|---|---|
| Client Authenticator | Client Id and Secret |
| Client Secret | Generated by Keycloak — copy immediately and store in a secret manager |

**Never commit the client secret to git.** In production, store it in Azure Key Vault or App Service environment variables.

## 3. Add a Protocol Mapper to Emit Realm Roles in the ID Token

BookStack reads role information from the **ID token**. Keycloak does **not** include `realm_access.roles` in the ID token by default — only in the access token. A Protocol Mapper on the `bookstack` client is therefore required to project realm roles into a top-level claim of the ID token that BookStack can consume.

**Path:** Keycloak console → target realm → **Clients → bookstack → Client scopes** tab → click the `bookstack-dedicated` scope → **Mappers** tab → **Configure a new mapper** → choose **By configuration** → **User Realm Role**.

| Field | Value |
|---|---|
| Name | `realm roles to id token` |
| Realm Role prefix | *(leave empty)* |
| Multivalued | **ON** |
| Token Claim Name | `roles` |
| Claim JSON Type | String |
| Add to ID token | **ON** |
| Add to access token | ON |
| Add to userinfo | ON |

Click **Save**.

With this mapper in place, the ID token returned to BookStack will include a top-level `roles` array, e.g.:

```json
{
  "iss": "http://host.docker.internal:8080/realms/<realm>",
  "sub": "...",
  "email": "user@example.com",
  "name": "User Display Name",
  "roles": ["platform-admin", "offline_access", "uma_authorization", "default-roles-<realm>"]
}
```

BookStack's OIDC config (`OIDC_GROUPS_CLAIM=roles`) matches this claim; if any value in the array matches a BookStack role's `external_auth_id`, that role is assigned to the user.

## 4. Expected Token Claims

BookStack relies on the following claims in the ID token:

| Claim | Expected value | Source |
|---|---|---|
| `iss` | `http://host.docker.internal:8080/realms/<realm>` | Generated by Keycloak — must exactly match the `OIDC_ISSUER` value configured in BookStack |
| `roles` | Top-level array containing `platform-admin` when the user has that realm role | Emitted by the User Realm Role Protocol Mapper configured in section 3 |
| `name` | User display name | Default Protocol Mapper (present on all Keycloak clients) |
| `email` | User email | Default Protocol Mapper |

To verify, decode a token returned by Keycloak (for example via [jwt.io](https://jwt.io)) and confirm the claims are present. On the BookStack side, the simplest way to inspect the ID token as BookStack actually receives it is to set `OIDC_DUMP_USER_DETAILS=true` in the BookStack config, log in, then read `/config/log/bookstack/laravel.log` inside the BookStack container. Remove `OIDC_DUMP_USER_DETAILS` once verification is complete; it writes full user details to the application log.

## 5. Verification

```bash
# 1. Discovery endpoint reachable (works from both browser and container perspectives)
curl -s http://host.docker.internal:8080/realms/<realm>/.well-known/openid-configuration | head

# 2. Confirm the `issuer` field exactly matches the OIDC_ISSUER configured on BookStack
curl -s http://host.docker.internal:8080/realms/<realm>/.well-known/openid-configuration \
  | grep -o '"issuer":"[^"]*"'
```

After configuration, sign in to BookStack via the "Keycloak" button and confirm:

1. The SSO flow completes and you land on the BookStack home page.
2. Users with `platform-admin` see the "Create" / "Edit" controls in BookStack.
3. Users without `platform-admin` have a read-only view.
4. Logging out of the OSP admin portal ends the BookStack session (open BookStack in a separate tab and refresh after adminui logout).

## 6. Production Replacement Checklist

Before going live, replace every occurrence of `http://host.docker.internal:6875` with the production FQDN:

- Root URL
- Home URL
- Valid redirect URIs
- Valid post logout redirect URIs
- Web origins
- Backchannel logout URL

And replace the Keycloak URL (`http://host.docker.internal:8080`) with the production Keycloak FQDN in:

- `OIDC_ISSUER` on the BookStack side

The production deployment topology (dedicated domain, subpath, or subdomain) should be confirmed with the customer — it affects TLS, reverse-proxy rules, and the exact values above.

## Troubleshooting

| Symptom | Likely cause | Resolution |
|---|---|---|
| "Invalid issuer" / SSO fails | `OIDC_ISSUER` on BookStack does not match Keycloak's `iss` claim | Ensure hostnames and ports match exactly; `localhost` and `host.docker.internal` are different |
| SSO redirects loop endlessly | Cookie domain mismatch — the user accessed BookStack via a different hostname than configured in `APP_URL` | Always use the same hostname in both BookStack's `APP_URL` and browser access |
| `platform-admin` users still appear as Viewer in BookStack | Role mapping missing, `OIDC_GROUPS_CLAIM` wrong, or the Protocol Mapper isn't emitting the claim into the ID token | Confirm the User Realm Role Protocol Mapper (section 3) has `Token Claim Name=roles` and `Add to ID token=ON`. On the BookStack side, `OIDC_GROUPS_CLAIM=roles` and Admin role `external_auth_id='platform-admin'` |
| `roles` claim is absent from the ID token | The Protocol Mapper was created on the wrong scope or `Add to ID token` is off | The mapper must live on the `bookstack` client's **dedicated** scope (not a generic realm-wide scope). Re-check each checkbox in the mapper configuration |
| BookStack session remains active after adminui logout | Backchannel Logout not functioning | Verify the Backchannel Logout URL is correct and BookStack's plugin handles it; otherwise switch to Front-channel Logout |
